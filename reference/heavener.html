heavener: This is what happens when you can't afford EDR licenses

edr-internals reverse-engineering windows-internals
Jun 26, 2026 · 20 min read

heavener is a modular EDR emulation engine for Windows that loads real detection logic extracted from commercial endpoint security products — their ML models, compiled YARA rulesets, and behavioral scripts — and evaluates it against live or replayed Windows telemetry. The user picks a vendor module and gets the exact verdicts the production EDR would produce.

Four vendor modules currently implemented: SentinelOne, Cortex XDR, CrowdStrike, and Sophos. Each loads genuine vendor artifacts and runs them through the same interface. Modules can be hot-swapped at runtime through the IPC client without restarting the engine.

Architecture

Six major layers connected by a clean data flow: telemetry comes in from the kernel driver and ETW providers, gets normalized into a uniform event schema, passes through enrichment and correlation, hits the active vendor module, and alerts flow out to a SOC-style web console.

Every event is a BehavioralEvent carrying a typed header and a std::variant payload over 19 concrete data structs. The taxonomy covers 25 event types:

enum class EventType : std::uint16_t
{
    ProcessCreate = 1, ProcessExit, ThreadCreate, ImageLoad,
    FileCreate, FileWrite, FileDelete, FileRename,
    RegistryCreateKey, RegistrySetValue, RegistryDeleteKey, RegistryDeleteValue,
    NetworkConnect, DnsQuery, DriverLoad, LdapQuery, WmiOperation,
    UserAccountCreated, ScheduledTaskCreated, ScriptExecution,
    BehavioralIndicator, ProcessHandleAccess, EtwTiEvent,
    NamedPipeCreate, FileSetBasicInfo,
};

The EventPipeline runs on a single worker thread — detection rules care about ordering. Events arrive via lock-free submit() and are drained in batches:

void worker_loop(std::stop_token token) noexcept
{
    while (m_running.load(std::memory_order_acquire))
    {
        std::deque<BehavioralEvent> batch;
        {
            std::unique_lock lock(m_queue_mutex);
            m_queue_cv.wait(lock, [&] {
                return !m_queue.empty()
                    || !m_running.load(std::memory_order_acquire);
            });
            batch.swap(m_queue);
        }
        for (auto& evt : batch)
            process_event(evt);
    }
}

Each process_event follows a strict three-stage pipeline: update the ProcessModel, run the ModifierEngine to attach aggregate signals, then feed the event to the active vendor module.

The Kernel Driver

heavendrv — a Windows kernel minifilter driver (altitude 370000) capturing:

  • Process/thread lifecycle via PsSetCreateProcessNotifyRoutineEx and PsSetCreateThreadNotifyRoutineEx, with kernel-captured call stacks
  • File operations: creates, writes, deletes, renames, named-pipe creation, SetBasicInfo for timestomping, FSCTL_SET_REPARSE_POINT for symlink abuse
  • Registry operations via CmRegisterCallbackEx: key creation, value sets with raw data up to 4KB, key/value deletion
  • Cross-process handle access via ObRegisterCallbacks: access mask and call stack frames

The wire protocol uses packed C structs with overloaded fields discriminated by event type:

typedef struct _HV_EVENT_HEADER {
    USHORT     Version;
    USHORT     Type;
    ULONG      Size;
    ULONGLONG  Timestamp;
    ULONG      Pid;
    ULONG      Tid;
    ULONG      ExtraPid;
    ULONG      U32;
    ULONGLONG  U64a;
    ULONGLONG  U64b;
    USHORT     VarCount;
} HV_EVENT_HEADER;

Variable-length fields (UTF-16LE strings, call stack frames) follow as length-prefixed blobs. Records capped at 64KB. The driver captures the creating process for each new process, not just the parent PID — parent PID spoofing detection.

ETW

Seven providers subscribed:

  • AMSI — full script content scanning (Powershell, VBScript, JScript)
  • WMI-Activity — WMI method calls and event subscriptions
  • LDAP — search filter strings for sensitive AD attribute queries
  • DNS — resolution events
  • Kernel providers (process, file, registry, network) — fallback when driver isn't loaded
  • ETW Threat Intelligence — PPL-protected, cross-process memory ops (ALLOCVM_REMOTE, PROTECTVM_REMOTE)

The ETW TI provider requires a separate PPL-AM collector service (heaventi-svc) backed by an ELAM driver.

The Process Model

Live in-memory process tree. Each node carries everything a detection rule could need:

struct ProcessInfo
{

    std::uint32_t  pid{};
    std::uint32_t  ppid{};
    std::wstring   name;
    std::wstring   command_line;
    std::wstring   image_path;
    std::uint64_t  create_time{};
    std::string    integrity_level;
    std::uint32_t  session_id{};
    std::wstring   user_sid;
    std::wstring   file_description;
    std::wstring   original_filename;
    std::wstring   company_name;
    std::string    cert_publisher;
    std::wstring   active_content_file;

    bool           is_remote_group{false};
    bool           has_injected_thread{false};
    std::uint32_t  injected_by_pid{};
    bool           was_opened_cross_process{false};
    std::uint32_t  opened_by_pid{};
    bool           opened_for_vm_read{false};
    bool           opened_for_vm_write{false};
    ProcessInfo*   model_parent{};
    std::unordered_map<std::uint32_t, std::string> extension_values;
};

Dual-indexed (PID for live, UID for historical). extension_values is a per-process dictionary vendor rules can read/write across events.

The Module System

Every vendor module implements a single C++ interface:

class IEdrModule
{
public:
    virtual std::string_view name() const noexcept = 0;
    virtual std::string_view version() const noexcept = 0;
    virtual ModuleCaps capabilities() const noexcept = 0;
    virtual std::expected<void, ModuleError>
        initialize(const ModuleConfig& cfg) noexcept = 0;
    virtual void shutdown() noexcept = 0;
    virtual std::expected<ScanResult, ModuleError>
        scan_file(std::span<const std::byte> data,
                  std::string_view file_path) noexcept = 0;
    virtual void on_event(const BehavioralEvent& evt,
                          const ProcessModel& model) noexcept;
    virtual std::vector<Alert> drain_alerts() noexcept;
};

Modules can be swapped at runtime through the IPC client without stopping telemetry collection.

Vendor Coverage

SentinelOne

Static ML (8 XGBoost models) + YARA + behavioral engine (203 Lua scripts consuming 19 event types through two dispatch mechanisms: event-class system and registerOSEventFilter with 137 OS-filter callbacks). Guard functions avoid entering the Lua VM for events the rule won't care about.

Cortex XDR

Three independent detection layers: YARA (36 rule sets), XGBoost ML, and a CLIPS expert system (~7,967 production rules, 6,448 templates, 581 functions, 1,502 globals). Full ECMAScript-compatible regex engine required for rule compatibility. Each event translates to a slot-complete CLIPS fact — a process-start fact alone carries 27 fields.

ETW-Ti path does stack enrichment with shellcode detection, syscall provenance classification (direct/indirect/normal), and byte-patch detection for EDR unhooking.

CrowdStrike

Static-only: XGBoost ML model. Behavioral detection embedded in channel files — active work on reversing the parsing/import logic.

Sophos

Two ML models (PE + documents) + traditional signature scanner. "Worst one wins" verdict combination.

The Replay System

Capture modes (--etw-capture, --driver-capture) record live events to JSON. --replay mode loads a JSON file and pushes every event through the full pipeline. Decouples testing from kernel driver and ETW.

Alerts and the Console

Alert IDs are deterministic FNV-1a hashes (idempotent replay). Alerts written as JSONL to an outbox directory, forwarded via a separate Go binary (heavener-forwarder) that POSTs gzip-compressed NDJSON batches with at-least-once delivery.

Web console stack: ClickHouse + Postgres + Redis + Go REST API + React SPA + Caddy. Features alerts timeline, investigation view, fleet view, and Rule Atlas (searchable database of all vendor behavioral rules).

Limitations

Everything runs what was extracted locally from agent binaries. Every vendor ships significant detection logic to the cloud — reputation lookups, cloud-side ML, threat intelligence correlations, dynamic analysis verdicts. That entire layer is a black box. heavener will never be a perfect replica, but it's the closest you can get without running the real agent.
